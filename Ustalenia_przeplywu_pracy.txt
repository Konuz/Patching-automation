Ustalenia do pełnej wersji skryptu patchującego wiele VM:

Docelowo nie utrzymujemy osobnego trybu wyboru update’ów dla jednej VM. Obecny tryb jednej VM był tylko etapem walidacji mechanizmów GuestOps/WUA. Pełna wersja ma być projektowana pod wiele VM.

Główny przepływ:

1. **Discovery**
   Skrypt uruchamia agenta na każdej VM i zbiera:
   - nazwę VM,
   - hostname z systemu,
   - dostępne aktualizacje z pełnymi tytułami, KB, `UpdateID`, `RevisionNumber`,
   - informację o `pending reboot` przed startem,
   - wykryte role/ryzyka: SQL, Exchange, Failover Cluster i ewentualnie inne role serwerowe.

2. **Twarde odrzucenie VM z klastrem**
   Jeśli skrypt wykryje Failover Cluster, VM nie jest patchowana automatycznie. Trafia do listy pominiętych z komunikatem:

   ```text
   Skipped: Failover Cluster detected. Please update manually one by one.
   ```

   SQL/Exchange mogą być oznaczane jako role wysokiego ryzyka obok nazwy VM, ale decyzja o ich automatycznym patchowaniu wymaga osobnych zasad. Failover Cluster na pewno ma być blokadą.

3. **Wybór update’ów nie per VM, tylko zbiorczo**
   Przy wielu VM nie używamy wyboru typu `1,2,3` ani `A`. Zamiast tego skrypt grupuje dostępne aktualizacje i pokazuje je jako checkboxy:

   ```text
   Available update groups

   [x] KB5060842 - 2026-06 Cumulative Update for Windows Server
       Applies to: 14 VM

   [x] KB5058392 - 2026-06 .NET Framework Cumulative Update
       Applies to: 9 VM

   [x] KB890830 - Windows Malicious Software Removal Tool
       Applies to: 12 VM

   [ ] KB5060821 - Preview Cumulative Update
       Applies to: 3 VM
   ```

   Użytkownik spacją przełącza `[x]` / `[ ]`. Do widoku człowieka używamy KB i pełnej nazwy aktualizacji, ale technicznie plan musi zapisywać `UpdateID + RevisionNumber`, bo KB i tytuł nie są wystarczająco jednoznaczne.

4. **Polityka domyślna**
   Skrypt może wstępnie zaznaczyć update’y według reguł:
   - zaznacz cumulative/security/critical/update rollup,
   - zaznacz Windows Malicious Software Removal Tool
   - pomiń preview,
   - pomiń drivers,
   - pomiń feature updates,
   - pomiń optional/browse-only, jeśli da się to wiarygodnie wykryć.

   Użytkownik może zmienić checkboxy przed zatwierdzeniem.

5. **Plan per VM po wyborze**
   Po zaakceptowaniu grup update’ów skrypt pokazuje finalny plan per VM, już tylko do weryfikacji:

   ```text
   Patch plan

   VM01
   Role flags: SQL
   Selected:
   - KB5060842 - 2026-06 Cumulative Update for Windows Server
   - KB5058392 - 2026-06 .NET Framework Cumulative Update

   VM02
   Role flags: none
   Selected:
   - KB5060842 - 2026-06 Cumulative Update for Windows Server

   VM03
   Skipped: Failover Cluster detected. Please update manually one by one.
   ```

6. **Finalne zatwierdzenie**
   Po podglądzie planu skrypt pyta:

   ```text
   Proceed with this plan? [Y/N]
   ```

   Dopiero po tym zaczyna instalację.

7. **Apply**
   Instalacja działa według zatwierdzonego planu:
   - limit równoległości, np. `ThrottleLimit 5`,
   - każda VM instaluje tylko update’y pasujące do zatwierdzonego `UpdateID + RevisionNumber`,
   - przed instalacją VM może ponownie zrobić search i dopasować update’y z planu,
   - wynik zapisywany per VM: installed, failed, reboot required, skipped.

8. **Raport końcowy**
   Na końcu raport:
   - VM spatchowane,
   - VM bez aktualizacji,
   - VM pominięte,
   - VM wymagające rebootu,
   - VM z błędami,
   - VM odrzucone przez Failover Cluster.

Kluczowa decyzja projektowa: pełna wersja ma mieć oddzielone etapy **discovery → wybór grup update’ów → plan per VM → finalne zatwierdzenie → apply**, zamiast ręcznego wyboru indeksów aktualizacji per pojedyncza VM.
